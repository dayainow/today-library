# Harness Meta Skill

이 스킬은 프로젝트에 맞는 에이전트 팀과 스킬을 설계, 생성, 점검, 진화시키는 메타 스킬이다.

## 원칙

1. 에이전트 정의는 `.claude/agents/{name}.md`에 둔다.
2. 실행 절차와 오케스트레이션은 `.claude/skills/{skill-name}/SKILL.md`에 둔다.
3. 새 세션 포인터는 `CLAUDE.md`에 최소한으로만 기록한다.
4. 중간 산출물은 `_workspace/`에 남겨 후속 실행과 감사가 가능하게 한다.
5. `.claude/commands/`는 만들지 않는다.

## Phase 0: 현황 감사

1. `.claude/agents/`, `.claude/skills/`, `CLAUDE.md`, `AGENTS.md`를 확인한다.
2. 신규 구축, 기존 확장, 운영/유지보수 중 어떤 상황인지 분류한다.
3. 에이전트 목록과 오케스트레이터 스킬의 팀 구성이 어긋나는지 확인한다.

## Phase 1: 도메인 분석

1. 사용자 요청의 도메인과 반복 작업 유형을 파악한다.
2. 생성, 검증, 편집, 분석, 배포, 운영 중 어떤 전문성이 필요한지 나눈다.
3. 코드베이스 구조와 실제 스택을 먼저 읽는다.

## Phase 2: 팀 아키텍처 선택

기본은 에이전트 팀이다. 둘 이상의 전문가가 서로 발견을 공유하거나 계약을 맞춰야 하면 팀으로 설계한다.

대표 패턴:

- 파이프라인: 분석 -> 설계 -> 구현 -> 검증처럼 순차 의존이 클 때.
- 팬아웃/팬인: 여러 관점의 병렬 조사나 리뷰 후 통합할 때.
- 전문가 풀: 입력 유형별로 필요한 전문가만 선택할 때.
- 생성-검증: 생성자와 검증자가 피드백 루프를 돌 때.
- 감독자: 작업량이 가변적이고 동적 분배가 필요할 때.
- 계층적 위임: 큰 도메인이 자연스럽게 하위 영역으로 나뉠 때.

## Phase 3: 에이전트 정의 생성

각 에이전트 파일에는 다음 섹션을 포함한다:

- 핵심 역할
- 작업 원칙
- 입력/출력 프로토콜
- 팀 통신 프로토콜
- 에러 핸들링
- 협업

## Phase 4: 스킬 생성

각 스킬은 frontmatter의 `name`, `description`을 반드시 포함한다. description에는 초기 실행 키워드뿐 아니라 "수정", "보완", "재실행", "업데이트", "이전 결과 기반" 같은 후속 트리거를 포함한다.

## Phase 5: 오케스트레이션

오케스트레이터 스킬은 다음을 명시한다:

- 실행 모드: 에이전트 팀, 서브 에이전트, 또는 하이브리드.
- 에이전트 구성표.
- `_workspace/` 기반 데이터 흐름.
- 실패/타임아웃/계약 충돌 처리.
- 정상 흐름과 에러 흐름 테스트 시나리오.

## Phase 6: 검증

1. 모든 에이전트와 스킬이 올바른 위치에 있는지 확인한다.
2. frontmatter를 확인한다.
3. 오케스트레이터의 팀 구성과 실제 파일이 일치하는지 확인한다.
4. should-trigger와 should-not-trigger 예시를 만들어 description 경계를 점검한다.

## Phase 7: 진화

사용자가 하네스 수정, 역할 추가, 워크플로우 변경, 검증 강화, 트리거 누락을 말하면 관련 에이전트/스킬을 갱신하고 `CLAUDE.md` 변경 이력에 남긴다.

## 현재 프로젝트 하네스

이 workspace의 기본 하네스는 `app-delivery-orchestrator`다. 앱 개발, 웹사이트 제작, 기능 구현, 버그 수정, 프론트엔드/백엔드/QA 협업 작업에서는 먼저 해당 오케스트레이터를 읽는다.
